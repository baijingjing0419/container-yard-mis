# 集装箱码头堆场管理MIS系统 - 数据库设计文档

## 一、设计概述

本数据库设计完全基于集装箱码头堆场管理系统的 **9大核心数据类** 和 **9大作业过程**，采用MySQL关系型数据库，遵循第三范式(3NF)设计原则。

### 1.1 核心数据类与表的映射关系

| 数据类编号 | 数据类名称 | 对应数据库表 | 关联作业过程 |
|-----------|-----------|-------------|-------------|
| D0 | 集装箱主数据 | `containers_master` | 所有集装箱生命周期流程 |
| D1 | 海侧入场集装箱信息 | `sea_inbound_containers` | 海侧进箱作业(C)、海侧作业计划(U) |
| D2 | 海侧出场集装箱信息 | `sea_outbound_containers` | 海侧出场作业(C)、海侧作业计划(U) |
| D3 | 海侧码头出、入场信息 | `sea_terminal_io` | 海侧作业计划(C)、海侧进箱/出场作业(U) |
| D4 | 陆侧入场集装箱信息 | `land_inbound_containers` | 陆侧进箱作业(C)、陆侧作业计划(U) |
| D5 | 陆侧出场集装箱信息 | `land_outbound_containers` | 陆侧出场作业(C)、陆侧作业计划(U) |
| D6 | 陆侧闸口出、入场信息 | `gate_io_records` | 陆侧作业计划(C)、陆侧进箱/出场作业(U) |
| D7 | 场内集装箱信息 | `yard_container_inventory` | 堆存日常管理(C)、调箱/进出箱作业(U) |
| D8 | 堆场作业记录 | `yard_operation_records` | 场内调箱作业(C)、堆存/进出箱作业(U) |
| D9 | 场内调度指令信息 | `dispatch_orders` | 中控调度(C)、调箱/进出箱作业(U) |
| M1 | 箱位移动流水 | `container_move_logs` | 调箱/翻箱/归位作业(U) |

## 二、详细表结构设计

### 2.0 主数据表

#### containers_master (集装箱主数据表)
解耦集装箱固有物理属性与业务流程数据，消除 D1/D2/D4/D5/D7 之间的循环依赖。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| container_id | VARCHAR(20) | PK | 箱号，如 MSKU7892345 |
| container_type | VARCHAR(10) | NOT NULL | 箱型：20GP/40GP/40HQ/45HQ |
| tare_weight | DECIMAL(10,2) | | 皮重(kg) |
| owner_company | VARCHAR(100) | INDEX | 所属船公司/箱公司 |
| size_code | VARCHAR(10) | | 尺寸代码，如 22G1/42G1 |
| manufacture_date | DATE | | 制造日期 |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |

**设计意图**：原 `yard_container_inventory` 表强依赖 `sea_inbound_containers`，导致陆侧箱无法正常入账。引入主数据表后，所有业务表的 `container_id` 均指向 `containers_master`，海侧/陆侧箱均可独立创建。

### 2.1 基础支撑表

#### ships (船舶信息表)
存储码头运营涉及的所有船舶基础信息。

| 字段名 | 类型 | 约束 | 说明 |
|-------|------|------|------|
| ship_id | VARCHAR(20) | PRIMARY KEY | 船舶编号/航次标识 |
| ship_name | VARCHAR(100) | NOT NULL | 船舶名称 |
| ship_type | VARCHAR(20) | | 船舶类型 |
| ship_company | VARCHAR(100) | | 所属船公司 |
| ship_length | DECIMAL(8,2) | | 船长(米) |
| ship_capacity | INT | | 载箱量(TEU) |
| status | VARCHAR(20) | DEFAULT 'active' | 状态 |

#### yard_zones (堆场区域表)
定义堆场的物理区域划分。

| 字段名 | 类型 | 约束 | 说明 |
|-------|------|------|------|
| zone_id | VARCHAR(10) | PRIMARY KEY | 区域编号(A/B/C) |
| zone_name | VARCHAR(50) | NOT NULL | 区域名称 |
| zone_type | VARCHAR(20) | NOT NULL | 类型(import/export/transit) |
| total_slots | INT | NOT NULL | 总箱位数 |
| occupied_slots | INT | DEFAULT 0 | 已占用数 |
| max_tier | INT | DEFAULT 5 | 最大堆叠层数 |

#### yard_slots (箱位明细表)
记录每个具体箱位的状态和占用情况。

| 字段名 | 类型 | 约束 | 说明 |
|-------|------|------|------|
| slot_id | VARCHAR(20) | PRIMARY KEY | 箱位编号(如A-12-04) |
| zone_id | VARCHAR(10) | FK → yard_zones | 所属区域 |
| row_num | INT | NOT NULL | 排号 |
| col_num | INT | NOT NULL | 列号 |
| tier_num | INT | DEFAULT 1 | 层号 |
| slot_status | VARCHAR(20) | DEFAULT 'empty' | 状态 |
| current_container_id | VARCHAR(20) | | 当前存放箱号 |
| max_weight | DECIMAL(10,2) | | 最大承重 |
| slot_size | VARCHAR(10) | | 适用箱型 |
| version | INT | NOT NULL DEFAULT 0 | **乐观锁版本号**：UPDATE 时 WHERE version = :old_version, SET version = version + 1 |

> **乐观锁机制**：高并发预占箱位时，先 SELECT 当前 version，UPDATE 时比对 `WHERE slot_id = :id AND version = :old_version`，若 affected_rows = 0 则重试。

### 2.5 流水日志表

#### container_move_logs (M1: 箱位移动流水表)

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| log_id | INT | PK AUTO_INCREMENT | 日志ID |
| container_id | VARCHAR(20) | FK → containers_master, NOT NULL | 箱号 |
| from_slot_id | VARCHAR(20) | FK → yard_slots | 原位置 |
| to_slot_id | VARCHAR(20) | FK → yard_slots, NOT NULL | 新位置 |
| move_time | DATETIME | NOT NULL | 移动时间 |
| operator_name | VARCHAR(50) | | 操作人员 |
| operation_id | VARCHAR(30) | | 关联作业记录号 |
| equipment_id | VARCHAR(20) | | 作业机械编号 |
| remark | VARCHAR(200) | | 备注 |

> **分区策略**：按 `move_time` 字段 `TO_DAYS()` 按月分区（共 19 个分区 + MAXVALUE），与 `system_logs`、`gate_io_records`、`yard_operation_records` 保持一致。

### 2.6 冷热数据分离

以下 3 张高频流水表已引入 **按月分区策略**（`PARTITION BY RANGE (TO_DAYS(created_at))`）：

| 表名 | 分区键 | 分区数 | 说明 |
|------|--------|--------|------|
| `system_logs` | created_at | 19 + MAXVALUE | 操作日志按月分区 |
| `gate_io_records` | created_at | 19 + MAXVALUE | 闸口通行记录按月分区 |
| `yard_operation_records` | created_at | 19 + MAXVALUE | 作业记录按月分区 |

查询时 MySQL 会自动进行分区修剪（Partition Pruning），只扫描目标月份的分区。

## 三、视图设计

### 3.1 v_yard_container_full (场内集装箱综合视图)
整合集装箱台账、位置、船舶、残损等信息的综合查询视图。

```sql
CREATE VIEW v_yard_container_full AS
SELECT 
    yci.container_id, yci.container_type, yci.container_status,
    yci.current_slot_id, ys.zone_id, yz.zone_name,
    yci.entry_time, yci.expected_exit_time, yci.dwell_time_hours,
    yci.is_overdue, yci.overdue_days, yci.alert_level,
    s.ship_name, s.ship_company, sic.manifest_info, sic.damage_status

LEFT JOIN yard_slots ys ON yci.current_slot_id = ys.slot_id
LEFT JOIN yard_zones yz ON ys.zone_id = yz.zone_id
LEFT JOIN ships s ON yci.ship_id = s.ship_id
LEFT JOIN sea_inbound_containers sic ON yci.container_id = sic.container_id;
```
